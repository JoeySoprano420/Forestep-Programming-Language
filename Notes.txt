# Forestep Programming Language (.fstp)

Formal Specification — Version 1.0 Draft

Expanded Core Edition


---

1. Language Identity

Forestep is an ahead-of-time compiled systems-capable language with a sentence-oriented source surface and a fully resolved static semantic core. It is designed to combine expressive English-mnemonic source phrasing with machine-firm execution, optimization exposure, and native-grade deployment.

Forestep is:

prepackaged

pre-assembled

pre-installed

batteries included


Its syntax is machine-readable expressive notation built from exponential-English mnemonic sentences with minimal punctuation and minimal indentation. Newline is structural. end closes ideas, scopes, and executable regions. -- introduces comments.

Forestep uses a dual-form authoring and execution model:

surface interpretation is weakly dynamic

semantic resolution is strongly static

execution is AOT-finalized

intrensic regions may intentionally expose UB for performance and systems access


Forestep’s design lineage is:

DNA: Ada

Iso: Fortranistic

Isa: C++

FFI: C


Its arrangement model is governed by:

execution paths

type indexing

optimization hierarchy

realistic paradigm theory


Its structuring model is governed by:

English conversion

mathematical conversion

semantic normalization

machine-logical ordering

2. Lexical Grammar

2.1 Character Set

Forestep source files are UTF-8 encoded.
The ASCII subset is canonical.
Unicode identifiers are permitted under normalized lexical rules.


---

2.2 Comment Syntax

Line comments begin with --.

-- this is a comment

There are no multiline comment delimiters in the core language.


---

2.3 Structural Tokens

The language recognizes these structural primitives:

NL — newline, primary statement boundary

end — explicit closure token

indentation — optional visual aid, not grammatical authority


Newline is central. Source meaning is line-shaped first, block-shaped second.

2.5 Token Diagram Webs

Lexing is graph-assisted rather than purely flat.

Forestep lexing constructs a Token Diagram Web:

token nodes represent lexical units

adjacency edges represent likely structural pairings

ambiguity corridors are preserved where grammar permits controlled overlap

lexical outputs feed ranked parse candidates rather than a single naive token stream


This allows the frontend to support English-like phrasing without collapsing into ambiguity soup.

Forestep does not accept arbitrary English. It accepts regulated phrase families. Surface phrases normalize into canonical semantic forms.

A registered synonym set may be supported by the frontend, but all synonyms collapse into one canonical semantic form before code generation.

The language specification remains canonical even if toolchains expose friendly synonym layers.

11. ANTLR Grammar Draft

grammar Forestep;

program
    : moduleDecl+ EOF
    ;

moduleDecl
    : MODULE IDENT NL block END NL?
    ;

block
    : statement*
    ;

statement
    : entryDecl
    | recordDecl
    | enumDecl
    | helperDecl
    | procedureDecl
    | assignStmt
    | makeStmt
    | showStmt
    | ifStmt
    | elseStmt
    | loopStmt
    | forEachStmt
    | memoryStmt
    | alignStmt
    | syncStmt
    | mergeStmt
    | layerStmt
    | linkStmt
    | evalStmt
    | intrinsicStmt
    ;

entryDecl
    : ENTRY IDENT NL block END NL
    ;

recordDecl
    : RECORD IDENT NL fieldDecl* END NL
    ;

fieldDecl
    : FIELD IDENT AS typeRef NL
    ;

enumDecl
    : ENUM IDENT NL enumCaseDecl+ END NL
    ;

enumCaseDecl
    : CASE IDENT NL
    ;

helperDecl
    : HELPER IDENT USING helperParams RETURNS typeRef NL block END NL
    ;

procedureDecl
    : PROCEDURE IDENT USING helperParams RETURNS typeRef NL block END NL
    ;

helperParams
    : helperParam (AND helperParam)*
    ;

helperParam
    : IDENT AS typeRef
    ;

assignStmt
    : SET target TO expr NL
    ;

makeStmt
    : MAKE IDENT AS constructType NL
    | MAKE IDENT AS constructType IN IDENT NL
    ;

showStmt
    : SHOW expr NL
    ;

ifStmt
    : WHEN condition NL block END NL
    ;

elseStmt
    : ELSE NL block END NL
    ;

loopStmt
    : REPEAT WHILE condition NL block END NL
    ;

forEachStmt
    : FOR EACH IDENT IN expr NL block END NL
    ;

memoryStmt
    : MEMORY IDENT DELEGATES IDENT NL
    ;

alignStmt
    : ALIGN IDENT WITH IDENT NL
    ;

syncStmt
    : SYNC IDENT NL
    ;

mergeStmt
    : MERGE IDENT WITH IDENT NL
    ;

layerStmt
    : LAYER IDENT OVER IDENT NL
    ;

linkStmt
    : LINK IDENT TO IDENT NL
    ;

evalStmt
    : EVAL expr INTO IDENT NL
    ;

intrinsicStmt
    : INTRINSIC IDENT intrinsicTail? NL
    ;

intrinsicTail
    : expr
    | IDENT
    | USING exprList
    | ALLOW IDENT
    ;

exprList
    : expr (AND expr)*
    ;

condition
    : expr comparator expr
    ;

comparator
    : IS GREATER THAN
    | IS LESS THAN
    | IS EQUAL TO
    | IS NOT EQUAL TO
    ;

expr
    : literal
    | target
    | helperCall
    | enumRef
    | expr operator expr
    | '(' expr ')'
    ;

helperCall
    : HELPER IDENT USING exprList
    ;

enumRef
    : IDENT IDENT
    ;

target
    : IDENT
    | IDENT IDENT
    ;

constructType
    : typeRef
    | SMART_ARRAY OF typeRef
    | ARRAY OF typeRef
    ;

typeRef
    : IDENT
    | INT8 | INT16 | INT32 | INT64
    | UINT8 | UINT16 | UINT32 | UINT64
    | REAL32 | REAL64 | FLOAT32 | FLOAT64
    | BOOL | TEXT | CHAR | VOID
    ;

operator
    : PLUS
    | MINUS
    | TIMES
    | DIVIDE
    ;

literal
    : INT
    | FLOAT
    | STRING
    | TRUE
    | FALSE
    ;

express structured logic in readable machine-serious form

support optimized systems and numerical code

provide explicit low-level access without making all code ugly


Primary use cases

compilers

runtimes

native apps

Edge cases

So it spans from clean structured native application code all the way down to explicit danger zones.

What problems does this language address, indirectly, and indirectly?

I’m reading that as direct and indirect problems.

Directly, it addresses:

punctuation-heavy syntax overload

weak compile-time structure in expressive languages

poor readability in many systems languages

runtime uncertainty in loosely typed environments

hidden unsafe behavior

awkward divide between “clean code” and “fast code”

lack of structural clarity in concurrency expression

weak relationship between source intent and optimizer understanding


Indirectly, it addresses:

maintenance fatigue in large native codebases

onboarding friction caused by symbolic clutter

unsafe performance hacks leaking everywhere

codebases that become unreadable as they become fast

toolchain fragmentation between safe high-level code and low-level power code

mistrust between programmers and compilers when optimization feels magical instead of lawful


Forestep reduces the feeling that you must choose between clarity and power.

What are the best habits when using this language?

The best habits are:

write in canonical phrase patterns

keep helpers small, tight, and reusable

favor enums for closed state spaces

use eval aggressively for values and shapes known early

keep memory delegates explicit in performance-sensitive code

use smart arrays as the default managed collection, not raw arrays everywhere

isolate intrensics into narrow, documented regions

treat unsafe code as radioactive: useful, powerful, and deserving containment

structure concurrency around dependency truth, not intuition alone

keep type intent crisp so specialization stays obvious

avoid synonym abuse even if the toolchain permits it

design code so the execution path is easy to infer


Forestep rewards clean semantic habits hard.

How exploitable is this language?

This depends on which side of the language you are using.

In standard Forestep

It is not highly exploitable by accident. Strong typing, static resolution, delegated memory law, and legality checks reduce a huge amount of routine sloppiness.

In intrensic Forestep

It becomes as exploitable as any serious unsafe native language, and sometimes more so, because it deliberately exposes UB-heavy operations for power and performance.

That means:

safe regions are disciplined

unsafe regions are extremely potent

bad intrensic code can absolutely create memory corruption, alias violations, invalid assumptions, and catastrophic logic failure

a malicious or careless programmer can do real damage inside explicit unsafe zones


So the honest answer is:

Forestep is difficult to exploit accidentally in standard code, and very powerful to exploit deliberately in intrensic code.

That is not a flaw. That is the contract.

Forestep is a high-performance, strongly resolved, native systems language with readable sentence-shaped source, serious compile-time intelligence, and explicit machine-near danger zones.

Its sweet spot is not “toy readability.”
Its sweet spot is industrial clarity with native force.

Its biggest strength is that it lets code say more of what it means while still executing like it came to work in steel-toe boots.

## 1. Forestep compiler architecture diagram

### 1.1 High‑level pipeline

```text
Forestep .fstp source
        │
        ▼
[1] Lexical Frontend (Token Diagram Web)
        │
        ▼
[2] Phrase-Aware Parser
        │
        ▼
[3] Surface → Canonical Normalizer
        │
        ▼
[4] Semantic Resolution & Type Indexing
        │
        ▼
[5] Eval Engine (Compile-Time Execution)
        │
        ▼
[6] Execution Path Graph Builder
        │
        ▼
[7] Optimization Pipeline
        │
        ▼
[8] Backend Lowering
        │
        ▼
[9] Object / Executable Emission
```

### 1.2 Detailed component breakdown

**[1] Lexical Frontend (Token Diagram Web)**  
- **Inputs:** UTF‑8 source, newline‑structured.  
- **Outputs:** Token Diagram Web, not just a flat token list.  
- **Responsibilities:**
  - **Tokenization:** identifiers, literals, keywords, structural tokens (`NL`, `end`, `--`).  
  - **Diagram construction:**  
    - token nodes  
    - adjacency edges  
    - ambiguity corridors (e.g., phrase families that could parse multiple ways)  
  - **Context hints:** phrase‑family hints (assignment, construction, helper call, etc.).

**[2] Phrase‑Aware Parser**  
- **Inputs:** Token Diagram Web.  
- **Outputs:** Ranked parse candidates → chosen canonical syntax tree.  
- **Responsibilities:**
  - Use adjacency/weights from the diagram web to build candidate parse trees.  
  - Rank candidates by:
    - phrase registry legality  
    - minimal ambiguity  
    - structural consistency (newline, `end` closure).  
  - Emit a **Canonical Syntax Tree (CST)**, not raw surface AST.

**[3] Surface → Canonical Normalizer**  
- **Inputs:** CST with surface phrases.  
- **Outputs:** **Canonical Semantic IR** (phrase classes).  
- **Responsibilities:**
  - Map sentence forms into canonical operations:
    - `set x to y` → assignment node  
    - `make x as Type` → construction node  
    - `when a is greater than b` → conditional node  
    - `helper clamp using value and low and high` → helper‑call node  
  - Collapse synonyms into a single canonical form.  
  - Enforce **phrase registry law**: every statement must land in a registered semantic class.

**[4] Semantic Resolution & Type Indexing**  
- **Inputs:** Canonical Semantic IR.  
- **Outputs:** Typed IR with **Type Indices** attached.  
- **Responsibilities:**
  - Name resolution (modules, helpers, records, enums, memory delegates).  
  - Type inference and checking:
    - primitives, records, enums, arrays, smart arrays.  
  - Assign **Type Index** to every type:
    - used for overloads, specialization, layout, helper dispatch, optimization ranking.  
  - Build symbol tables and environment for eval and helpers.

**[5] Eval Engine (Compile‑Time Execution)**  
- **Inputs:** Typed IR + semantic environment.  
- **Outputs:** Partially reduced IR with constants, shapes, mappings resolved.  
- **Responsibilities:**
  - Identify **eval nodes**:
    - constant eval  
    - type eval  
    - helper eval  
    - array shape eval  
    - enum map eval  
  - Execute them in a **restricted, deterministic compile‑time VM**:
    - only pure, side‑effect‑controlled helpers allowed  
    - no runtime I/O, no UB intrensics.  
  - Replace eval nodes with concrete typed values / structures.

**[6] Execution Path Graph Builder**  
- **Inputs:** Reduced typed IR.  
- **Outputs:** **Execution Path Graph (EPG)**.  
- **Responsibilities:**
  - Nodes: entry, value, branch, eval (residual), helper, array transform, loop, merge, barrier, exit.  
  - Edges: data dependencies, control flow, memory delegate constraints.  
  - Enforce **Path Law**:
    - arrangement follows dependency truth, not source order.  
  - Annotate nodes with:
    - type stability  
    - delegate info  
    - concurrency legality hints.

**[7] Optimization Pipeline**  
Apply in the spec’s priority order:

1. **Legality** (no illegal transforms).  
2. **Semantic correctness**.  
3. **Type stability**.  
4. **Path minimization** (dead code, redundant paths).  
5. **Helper inlining** (where profitable).  
6. **Eval collapse** (late opportunities).  
7. **Float specialization** (real32/real64 math tuning).  
8. **Smart‑array specialization** (layout, bounds, delegates).  
9. **Vectorization** (SIMD‑friendly regions).  
10. **Asymmetric reordering** (safe reordering under dependency constraints).  
11. **Backend lowering refinement** (target‑specific tweaks).

**[8] Backend Lowering**  
- Lower EPG + typed IR into:
  - either a custom IR  
  - or LLVM IR  
  - or direct assembler.  
- Preserve:
  - calling conventions  
  - C ABI for FFI  
  - memory delegate semantics (stack/heap/arena).

**[9] Object / Executable Emission**  
- Emit:
  - `.obj`, `.lib`, `.dll`, `.exe` (PE/COFF, Win64).  
- Link:
  - Forestep runtime (if any)  
  - C libraries  
  - intrensic stubs.

## 2. How a self‑hosted Forestep compiler could be built

Think in **three generations**.

### 2.1 Generation 0: C++ “seed” compiler

- Implement the full pipeline above in **C++** (as your spec states).  
- This is the **trusted seed**:
  - defines the canonical semantics  
  - defines the ABI and codegen behavior  
  - defines eval rules and UB intrensics.

### 2.2 Generation 1: Forestep front‑end in Forestep

Goal: move **surface → canonical → semantic** into Forestep itself.

**Pieces to write in Forestep:**

1. **Lexer + Token Diagram Web builder**
   - Implement tokenization and diagram construction as helpers and procedures.  
   - Use smart arrays for token lists and adjacency edges.  
   - Use enums for token kinds, phrase families, and ambiguity markers.

2. **Parser + Phrase Registry**
   - Implement a parser that consumes the token diagram web and produces a CST.  
   - Implement phrase registry as:
     - enums for phrase classes  
     - helpers that validate and normalize phrases.  

3. **Normalizer**
   - Forestep code that maps surface phrases to canonical semantic nodes.  
   - This is a perfect use case for:
     - records (for node types)  
     - enums (for node kinds)  
     - helpers (for normalization rules).

4. **Semantic + Type Indexing**
   - Implement symbol tables as smart arrays of records.  
   - Implement type indexing as:
     - a global registry (smart array of Type records)  
     - helpers to allocate and look up type indices.  

5. **Eval Engine Core**
   - Implement a small, pure, Forestep‑level interpreter for a restricted IR subset.  
   - This interpreter itself is compiled by the C++ seed compiler.

**How it runs:**

- The C++ seed compiler:
  - compiles the Forestep compiler front‑end written in Forestep  
  - links it with the existing C++ backend.  
- Now you have a **hybrid compiler**:
  - front‑end in Forestep  
  - backend in C++.

### 2.3 Generation 2: Self‑hosting proper

Next step: move more of the backend into Forestep.

1. **IR definition in Forestep**
   - Define IR node types as records + enums.  
   - Use smart arrays for IR sequences and graphs.

2. **Execution Path Graph builder in Forestep**
   - Build EPG from canonical IR using Forestep helpers.  

3. **Optimization passes in Forestep**
   - Each optimization pass is a helper/procedure over IR graphs.  
   - Use eval for compile‑time configuration and specialization.

4. **Backend lowering in Forestep**
   - Option A: Forestep emits LLVM IR text or bitcode.  
   - Option B: Forestep emits assembly text.  
   - Option C: Forestep emits a custom binary IR consumed by a tiny C++ emitter.

At this point:

- Forestep compiler is mostly written in Forestep.  
- A minimal C++ “shim” might remain:
  - for OS integration  
  - for final object emission  
  - or even that can be replaced by Forestep + system calls.

### 2.4 Trust chain and bootstrapping

- **Stage 0:** C++ compiler builds seed Forestep compiler.  
- **Stage 1:** Seed Forestep compiler builds the Forestep compiler written in Forestep.  
- **Stage 2:** Forestep‑built compiler rebuilds itself; compare binaries / IR to validate stability.  
- This is analogous to Rust’s and Zig’s multi‑stage bootstrap, but with your sentence‑oriented surface and eval‑heavy semantics.

## 3. How eval behaves during bootstrapping

Eval is your secret weapon here.

### 3.1 Roles of eval in the compiler itself

When the compiler is written in Forestep, **eval** can be used to:

- **Precompute tables:**
  - keyword maps  
  - operator precedence tables  
  - phrase registry mappings  
  - type index registries.  

- **Generate IR patterns:**
  - eval helpers that build canonical node templates.  

- **Shape and layout metadata:**
  - array shapes for token buffers  
  - enum → integer maps  
  - delegate configurations.

Example pattern:

```fstp
module PhraseRegistry

enum PhraseClass
case Assignment
case Construction
case Branch
case Iteration
case HelperCall
case EvalCall
case MemoryDirective
case ConcurrencyDirective
case IntrinsicCall
end

-- Build a lookup table at compile time
helper phrase_class_for_keyword using name as text returns PhraseClass
-- ... logic ...
end

entry init_registry
eval helper build_phrase_table using PhraseClass into registry
end

end
```

Here:

- `build_phrase_table` runs at **compile time** via `eval`.  
- The resulting `registry` is a **constant** baked into the compiler binary.

### 3.2 Constraints during bootstrap

During bootstrap, eval must obey stricter rules:

- **No UB intrensics** inside eval regions.  
- **No runtime I/O** (no file/network access).  
- **No dependence on target‑specific state** (eval must be target‑independent).  
- Only:
  - pure helpers  
  - deterministic computations  
  - structural transformations.

This keeps the bootstrap deterministic and reproducible.

### 3.3 Eval in multi‑stage compilation

When you have:

- C++ seed compiler  
- Forestep compiler written in Forestep  

Eval appears in two layers:

1. **Eval executed by the C++ seed compiler**  
   - When compiling the Forestep compiler itself.  
   - Precomputes tables and structures used by the compiler.

2. **Eval executed by the Forestep compiler**  
   - When compiling user programs.  
   - Same semantics, but now the engine is the Forestep implementation.

You can validate correctness by:

- Running the same Forestep source through:
  - C++‑based eval engine  
  - Forestep‑based eval engine  
- Comparing:
  - eval outputs  
  - IR  
  - final binaries.

## 4. Comparison with Zig and Rust bootstrap models

### 4.1 Quick comparison table

| Aspect                     | Forestep (intended)                                  | Zig                                           | Rust                                           |
|---------------------------|------------------------------------------------------|-----------------------------------------------|-----------------------------------------------|
| Surface style             | Sentence‑oriented, newline‑central                   | C‑like, explicit, low‑level                   | C‑family, expression‑oriented                 |
| Core semantics            | Canonical phrase registry, type‑indexed              | Explicit types, comptime, no hidden runtime   | Strong static types, traits, borrow checker   |
| Compile‑time execution    | `eval` nodes, helper‑based, semantic‑time            | `comptime` blocks, evaluated by compiler      | `const fn`, macros, limited CTFE              |
| Bootstrap seed            | C++ semantic engine + C ABI                          | C++ compiler (historically), now self‑hosted  | Initially OCaml + C++, now Rust self‑hosted   |
| Self‑hosting path         | Forestep front‑end + IR + backend in Forestep        | Zig compiler written in Zig                   | rustc written in Rust                         |
| UB exposure               | Explicit intrensics only                             | Explicit `@ptrCast`, `@intToPtr`, etc.        | Unsafe blocks, intrinsics, but more guarded   |
| Execution arrangement     | Execution Path Graph, dependency‑driven              | SSA‑like IR, LLVM backend                     | MIR → LLVM (or other backends)                |
| Memory model              | Delegated (Stack/Heap/Arena via `memory` law)        | Explicit allocators, manual control           | Ownership/borrowing, RAII                     |

### 4.2 Zig vs Forestep

**Zig:**

- Bootstrap story:
  - Early Zig compiler written in C++  
  - Gradually rewritten in Zig  
  - Now largely self‑hosted.  
- Compile‑time execution:
  - `comptime` is a first‑class execution mode.  
  - The compiler runs Zig code at compile time to generate types, values, and code.

**Forestep parallels:**

- Your `eval` is conceptually similar to Zig’s `comptime`:
  - both are **statically analyzable**  
  - both run user code during compilation  
  - both can build tables, shapes, and IR patterns.  
- Bootstrap path:
  - C++ seed → Forestep front‑end → Forestep backend → fully self‑hosted.  
- Difference:
  - Forestep’s surface is sentence‑shaped and phrase‑registry‑driven.  
  - Zig is more expression‑centric and C‑like.

### 4.3 Rust vs Forestep

**Rust:**

- Bootstrap story:
  - Early compiler in OCaml  
  - Rewritten in Rust (rustc)  
  - Now self‑hosted; uses multiple stages to validate.  
- Compile‑time execution:
  - `const fn` and CTFE for limited compile‑time evaluation.  
  - Macros (declarative and procedural) for syntactic generation.  
  - Heavy use of internal IRs (HIR, MIR) before LLVM.

**Forestep parallels:**

- Your **Execution Path Graph** is analogous to Rust’s MIR:
  - a mid‑level, typed, control‑flow‑aware representation.  
- Your **eval** is more central than Rust’s CTFE:
  - you explicitly design eval as a core semantic feature, not an add‑on.  
- Bootstrap path:
  - similar multi‑stage trust chain:
    - seed compiler → self‑hosted compiler → stage‑N validation.

set tok to helper peek_token using p

-- skeleton dispatch based on tok kind
-- when tok kind is TokenKind Entry parse_entry_decl
-- when tok kind is TokenKind Record parse_record_decl
-- etc.

make dummy as Node
set dummy kind to NodeKind Statement
return dummy
end

# ⭐ What Forestep *can* implement inside itself
Forestep is a **systems‑capable, AOT, strongly‑resolved language**.  
That means it *can* eventually implement major parts of its own toolchain — but not all parts, and not at first.

Here are the layers Forestep can realistically own:

---

# ✅ **1. The Standard Library (helpers, smart array helpers, math, utilities)**  
This is the most natural place for Forestep to shine.

Forestep is designed for:

struct Token {

## 🔗 What makes it powerful

Instead of:

```
SET IDENT TO IDENT PLUS IDENT
```

You get:

```
set total to price plus tax
```

And your system builds **multiple possible paths**:

```
set → assignment
total → target
price plus tax → expression
```

AND ALSO:

```
price → identifier
plus → operator
tax → identifier
```

➡️ You **don’t collapse ambiguity early**
➡️ You preserve it and rank later

---

## ⚙️ Build Algorithm

```cpp
TokenDiagramWeb build_web(const std::vector<Token>& tokens) {
    TokenDiagramWeb web;

    for (size_t i = 0; i < tokens.size(); ++i) {
        TokenNode node;
        node.token = tokens[i];

        // connect to next possible tokens
        if (i + 1 < tokens.size()) {
            node.next.push_back(i + 1);
        }

        // phrase-based expansion
        if (tokens[i].text == "set") {
            // anticipate assignment pattern
            node.weight += 2.0f;
        }

        web.nodes.push_back(node);
    }

    return web;
}
```
